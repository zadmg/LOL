### 2.1 登录验证流程

```
    ↓ 通过

```

### 2.2 补丁点1: 0x0142CB

**功能**：跳过 API 网络验证调用

**原始指令**（6 字节 jz rel32）：
```
0F 84 0A 00 00 00    jz $+0x0A+6  (跳过 API 调用)
```

**修改后**（5 字节 jmp + 1 字节 nop）：
```
E9 0B 00 00 00 90    jmp $+0x0B+5  (强制跳过); nop
```

**关键细节**：
- jz 是 6 字节指令，jmp 是 5 字节指令
- jz rel32 的相对偏移是 0x0A（从 jz 指令末尾算起）
- jmp rel32 的相对偏移需要调整为 0x0B（从 jmp 指令末尾算起，比 jz 多 1 字节）
- 末尾补 1 个 0x90 (nop) 保持总长度 6 字节，避免指令错位

**为什么不能用 jz→jmp rel8**：
- 原始 jz 是 6 字节，jmp rel8 只有 2 字节，会留下 4 字节空洞
- 指令对齐要求保持 6 字节总长度

### 2.3 补丁点2: 0x016B43

**功能**：跳过授权数据验证

**原始指令**（6 字节 jz rel32）：
```
0F 84 84 00 00 00    jz $+0x84+6
```

**修改后**（6 个 nop）：
```
90 90 90 90 90 90    nop; nop; nop; nop; nop; nop
```

**为什么这里用 nop 而非 jmp**：
- 这里是"验证失败则跳转"的逻辑
- 用 nop 替换后，无论验证结果如何都继续执行（不跳转）
- 等于"忽略验证失败"，直接放行

### 2.4 补丁验证脚本

```python
# patch_beifen.py 核心逻辑
PATCHES = [
    {
        "offset": 0x0142CB,
        "orig":    bytes.fromhex("0F840A000000"),    # jz rel32 (6 字节)
        "patched": bytes.fromhex("E90B00000090"),    # jmp rel32(0x0B) + nop (6 字节)
    },
    {
        "offset": 0x016B43,
        "orig":    bytes.fromhex("0F8484000000"),    # jz rel32
        "patched": bytes.fromhex("909090909090"),    # 6 个 nop

```

## 3. MJTVPFeD.dat 三层嵌套结构

### 3.1 Overlay 提取

```
16.13.exe
└─ @ 0x121000: Overlay 开始
    ├─ [0x121000 - 0x121007]  8 bytes header: D6 F4 ED 60 29 E9 C8 94
    └─ [0x121008 - 0x13C7C7]  113,664 bytes RC4 加密数据
        └─ XOR keystream 解密 → MJTVPFeD.dat 明文 (113,664 bytes)
```

### 3.2 RC4 解密

**密钥流**：`rc4_keystream.bin`（113,664 bytes）

**解密公式**：
```python
plaintext = bytes(a ^ b for a, b in zip(encrypted, keystream))
```

**关键特征**：
- RC4 是流密码，加密和解密相同（XOR）
- 密钥流可重用，只要明文长度不变
- 修改 PE2 后只需重新 XOR 加密，密钥流不变

### 3.3 MJTVPFeD.dat 内部结构

```
MJTVPFeD.dat (113,664 bytes = 0x1BC00)
├─ [0x00000 - 0x0B9EF]  外层 EXE (47,856 bytes)
│   └─ MZ 头 @ 0x0
│       功能: 释放 PE1 和 PE2 到磁盘
│
├─ [0x0B9F0 - 0x0F1CF]  PE1: dbgcore.dll (14,272 bytes)
│   └─ MZ 头 @ 0xB9F0
│       类型: UPX 加壳的 DLL 劫持器
│       功能: 转发 dbgcore.dll 调用 + 加载 flyinput.dll
│
├─ [0x0F1D0 - 0x16BCF]  PE2: flyinput.dll (31,232 bytes)
│   └─ MZ 头 @ 0xF1D0
│       类型: UPX 加壳的主辅助 DLL
│       功能: TimerFunc 主循环 + 英雄按键模拟
│
└─ [0x16BD0 - 0x1BBFF]  附加数据 (20,528 bytes)
    ├─ Thawte CA 证书链 (Pikachu CA 证书固定)
    ├─ szOID_NESTED_SIGNATURE 格式签名
    └─ 配置表 (3 tokens + 签名协议)
```

### 3.4 证书固定（Certificate Pinning）

**目的**：仅信任嵌入的证书，忽略系统证书存储

**实现**：
- 使用 Pikachu CA 证书链
- szOID_NESTED_SIGNATURE 格式
- 用于认证 api2.xbzhan.com 通信

**对破解的影响**：
- 修改 PE2 后需要重新签名（或绕过证书检查）
- 实际操作中：保留原始证书数据，仅替换 PE2 区域

## 5. TimerFunc 主循环分析

### 5.1 函数信息

```
名称:    TimerFunc
RVA:     0x14020
文件偏移: 0x13420 (解壳后)
触发:    SetTimer 回调，每 1ms 执行
参数:    rcx=hWnd, rdx=uMsg(WM_TIMER), r8=idEvent, r9=dwTime
```

### 5.2 原始头部

```
0x14020: 56              push rsi
0x14021: 48 83 EC 20     sub rsp, 20h
0x14025: 49 89 D0        mov r8, rdx        ← TimerFunc+5 (后续代码开始)
...
```

### 5.3 工作流程

```
TimerFunc 被 SetTimer 每 1ms 调用
    ↓
检查 r8 (idEvent) 判断是哪个 timer
    ↓
case 1: Ashe (寒冰) - 原始英雄
case 2: (预留)
case 3: (预留)
case 4: Jinx (金克丝) - 移植的英雄
    ↓
检查 GetAsyncKeyState 判断玩家按键
    ↓
按固定时序发送 QWER (SendInput)
    ↓
返回
```

### 5.4 case 跳转表

```
case 1 (Ashe):
    检测空格键 → 发送 QWER 序列

case 4 (Jinx):
    检测空格键 → 发送 QWER 序列
    (克隆自 Ashe 模板，修改技能 ID)
```

### 5.5 盲发模式

**核心特征**：TimerFunc **不读游戏状态**（血量/CD/位置）

**工作方式**：
- 只检测玩家按键（GetAsyncKeyState，全局键盘状态）
- 按固定时序发送 QWER（SendInput，全局输入）
- 不知道游戏内发生了什么

**这是防检测设计的核心**：不读游戏内存 = 不暴露检测面

## 7. Jinx 英雄移植

### 7.1 移植思路

### 7.2 实现步骤


### 7.3 BST (二叉搜索树) 扩展

**BST 用途**：根据英雄 ID 查找对应函数地址

**FNV-1a 哈希**：
```python
def fnv1a_hash(string):
    hash = 0xCBF29CE484222325
    for c in string:
        hash ^= ord(c)
        hash = (hash * 0x100000001B3) & 0xFFFFFFFFFFFFFFFF
    return hash
```

**Jinx 的哈希值**：通过计算 "Jinx" 字符串的 FNV-1a 哈希，插入 BST

```

    ├─ case 2: jmp (预留)
    ├─ case 3: jmp (预留)
    └─ case 4: jmp Jinx 函数 (.newtext + 0x200)
```

## 8. 假点击功能实现

### 8.1 设计目标

迷惑反作弊，模拟人类鼠标微动，避免被检测为"无鼠标移动的脚本"

### 8.2 v1: 简单假点击（已废弃）

```
触发概率: 1/256
动作: 单步 ±8 像素瞬时跳跃
持续时间: 1ms
```

**问题**：过于简单，不像人类操作

### 8.3 v2: 8 步轨迹（有 bug）

```
触发概率: 1/128
动作: 8 步 ±3 像素平滑滑动
持续时间: 8ms (8 个 TimerFunc 周期)
状态机: 空闲 → 生成轨迹 → 执行 8 步 → 空闲
```

**致命 bug**：
1. **寄存器未保存**：破坏 TimerFunc 回调参数 (rcx/rdx/r8/r9)
2. **栈未对齐**：`sub rsp, 28h` (40 bytes) 不是 16 的倍数

**症状**：进入游戏后无任何反应（Jinx 功能也失效）

### 8.4 v3: 修复版（当前版本）

#### 8.4.1 修复内容

**Bug 1: 寄存器保存**

```asm
; v2 入口
push rsi
sub rsp, 20h
; 直接使用 rax/rbx/rcx/rdx/r8 ← 破坏 TimerFunc 参数!

; v3 入口
push rsi
sub rsp, 20h
push rax              ; ← 新增
push rcx              ; ← 新增
push rdx              ; ← 新增
push r8               ; ← 新增 (保存 idEvent!)
push r9               ; ← 新增
push rbx              ; ← 新增
```

**Bug 2: 栈对齐**

```asm
; v2
sub rsp, 28h    ; 40 bytes, 不对齐 16

; v3
sub rsp, 50h    ; 80 bytes = 40 INPUT + 32 shadow + 8 对齐
```

#### 8.4.2 完整函数结构

```asm
; === 入口 ===
push rsi
sub rsp, 20h           ; 原始 TimerFunc 头部
push rax/rcx/rdx/r8/r9/rbx  ; 保存寄存器

; === 读状态 ===
movzx eax, [fc_step]
cmp al, 0xFF
jne .execute           ; 执行中 → 跳到执行

; === 空闲: 概率触发 ===
rdtsc
and ecx, 0x7F          ; 1/128 概率
cmp ecx, 1
ja .done               ; 未触发 → done

; === 生成 8 步轨迹 ===
xor ebx, ebx           ; i = 0
.gen_loop:
    rdtsc
    mov ecx, eax
    and ecx, 7
    sub ecx, 3         ; dx = -3..+4
    lea rdx, [fc_dx]
    mov [rdx + rbx*4], ecx
    shr eax, 8
    and eax, 7
    sub eax, 3         ; dy = -3..+4
    lea rdx, [fc_dy]
    mov [rdx + rbx*4], eax
    inc ebx
    cmp ebx, 8
    jb .gen_loop
mov byte [fc_step], 0   ; 进入执行态
jmp .done

; === 执行轨迹步 ===
.execute:
lea rdx, [fc_dx]
mov ecx, [rdx + rax*4]  ; ecx = dx[step]
lea rdx, [fc_dy]
mov r8d, [rdx + rax*4]  ; r8d = dy[step]

; === 构造 INPUT 结构 ===
sub rsp, 50h            ; 80 bytes (栈对齐 16)
xor eax, eax
mov [rsp], eax           ; type = INPUT_MOUSE (0)
mov [rsp+8], ecx         ; dx
mov [rsp+0Ch], r8d       ; dy
mov dword [rsp+14h], 1   ; dwFlags = MOUSEEVENTF_MOVE
mov ecx, 1               ; cInputs
lea rdx, [rsp]           ; pInputs
mov r8d, 28h             ; cbSize = 40
call [SendInput]         ; RIP-relative IAT 调用
add rsp, 50h

; === 步进/重置 ===
movzx eax, [fc_step]
inc eax
cmp eax, 8
jb .keep
mov al, 0xFF             ; 完成，回到空闲
.keep:
mov [fc_step], al

; === 出口 ===
.done:
pop rbx/r9/r8/rdx/rcx/rax  ; 恢复寄存器
jmp TimerFunc+5           ; 返回原 TimerFunc
```

#### 8.4.3 关键偏移

```
假点击函数 RVA:    0x1E800 (.newtext + 0x1800)
状态变量 RVA:      0x1F000 (.newtext + 0x2000)
  fc_step:         0x1F000 (1 byte)
  fc_dx[8]:        0x1F004 (32 bytes)
  fc_dy[8]:        0x1F024 (32 bytes)
SendInput IAT:     0x17B08
TimerFunc 入口:     0x14020
TimerFunc 返回:     0x14025
```

#### 8.4.4 工作流程

```
TimerFunc 每 1ms 触发
    ↓
jmp 假点击函数 (0x1E800)
    ↓
保存寄存器
    ↓
检查 fc_step:
    ├─ 0xFF (空闲) → rdtsc 1/128 概率触发
    │   ├─ 触发 → 生成 8 步轨迹 → fc_step=0
    │   └─ 未触发 → done
    └─ 0-7 (执行中) → 读 dx/dy[step] → SendInput → step++
        └─ step≥8 → fc_step=0xFF
    ↓
恢复寄存器
    ↓
jmp TimerFunc+5 (继续原 TimerFunc 逻辑)
```

## 13. 踩坑记录

### 13.1 jz→jmp 的 rel32 调整

**问题**：jz(6字节)转jmp(5字节)时，相对偏移需要从 0x0A 调整为 0x0B

**原因**：
- jz rel32 的跳转目标 = jz 指令地址 + 6 + rel32
- jmp rel32 的跳转目标 = jmp 指令地址 + 5 + rel32
- 同样跳到同一目标，jmp 的 rel32 比 jz 大 1

**解决**：rel32 从 0x0A 改为 0x0B，末尾补 nop

### 13.2 假点击 v2 寄存器破坏

**问题**：进入游戏后无任何反应

**原因**：假点击函数破坏了 TimerFunc 的回调参数寄存器
- rcx = hWnd
- rdx = uMsg
- r8 = idEvent（**决定执行哪个 case**）
- r9 = dwTime

v2 直接使用这些寄存器做 rdtsc/and/lea/mov，导致 TimerFunc 后续代码读到错误的 r8，无法匹配任何 case

**解决**（v3）：入口 push 保存，出口 pop 恢复

### 13.3 假点击 v2 栈未对齐

**问题**：调用 SendInput 可能崩溃

**原因**：`sub rsp, 28h` (40 bytes) 不是 16 的倍数，x64 ABI 要求调用前 rsp 对齐 16

**解决**（v3）：`sub rsp, 50h` (80 bytes = 40 INPUT + 32 shadow + 8 对齐)

### 13.4 PE2 槽位大小限制

**问题**：假点击 v2 加壳后 31,744 bytes，超过槽位 31,232 bytes

**解决**：使用 `--ultra-brute` 压缩到 30,720 bytes，尾部补 512 bytes 的 0 填充

### 13.5 koa-connect ctx 泄漏

**问题**：早期尝试用 koa-connect 包装 Express 中间件导致 ctx 泄漏

**解决**：改用原生 Koa 重写，不包装 Express 中间件

### 13.6 Section name bytearray 扩展

**问题**：PE 节区名是 8 字节，直接赋值可能截断

**解决**：用 bytearray 扩展到 8 字节

### 13.7 mov 指令长度计算

**问题**：误判 mov 指令长度为 8 字节，实际是 9 字节

**解决**：用 Capstone 反汇编验证实际长度

## 14. dbgcore.dll 劫持机制深度解析

### 14.1 劫持原理

**DLL 劫持**（DLL Hijacking）利用了 Windows 的 DLL 搜索顺序。当 `LeagueClient.exe` 启动时，它需要加载 `dbgcore.dll`（Windows 崩溃转储组件）。Windows 的 DLL 搜索顺序是：

```
1. 应用程序所在目录          ← 游戏目录
2. System32 目录
3. 16-bit 系统目录
4. Windows 目录
5. 当前工作目录
6. PATH 环境变量目录
```

由于游戏目录优先级最高，只要在游戏目录放置一个伪造的 `dbgcore.dll`，`LeagueClient.exe` 就会优先加载它，而不是 `C:\Windows\System32\dbgcore.dll`。

### 14.2 伪造 dbgcore.dll 的完整工作流程

```
LeagueClient.exe 启动
    ↓
LoadLibrary("dbgcore.dll")
    ↓
Windows 搜索: 游戏目录\dbgcore.dll → 找到伪造的!
    ↓
DllMain(DLL_PROCESS_ATTACH) 被调用
    ↓
sub_180001994(hinstDLL, fdwReason=1, ...)
    ↓
sub_180001020(hinstDLL, a2=1)  ← 核心初始化函数
    │
    ├─ DisableThreadLibraryCalls(a1)
    │   └─ 禁用线程通知，减少 DLL 加载痕迹
    │
    ├─ GetSystemDirectoryA(Buffer, 0x104)
    │   └─ 获取 C:\Windows\System32 路径
    │
    ├─ lstrcatA(Buffer, "\\dbgcore.dll")
    │   └─ 拼接 = C:\Windows\System32\dbgcore.dll
    │
    ├─ LoadLibraryA(Buffer)
    │   └─ 加载真正的系统 dbgcore.dll
    │
    ├─ GetProcAddress(hLibModule, "MiniDumpReadDumpStream")
    │   └─ 保存真函数指针到 qword_180003158
    │
    ├─ GetProcAddress(hLibModule, "MiniDumpWriteDump")
    │   └─ 保存真函数指针到 qword_180003160
    │
    └─ LoadLibraryA("C:\\Users\\Public\\FlyPY\\flyinput.dll")
        └─ 加载辅助 DLL，完成注入
```

### 14.3 导出函数转发机制

伪造的 dbgcore.dll 只导出 3 个函数：

| 导出函数 | 地址 | 实现 |
|---------|------|------|
| MiniDumpReadDumpStream | 0x180001000 | `jmp [qword_180003158]` → 真函数 |
| MiniDumpWriteDump | 0x180001010 | `jmp [qword_180003160]` → 真函数 |
| DllEntryPoint | 0x180001ABC | 标准入口 |

**转发代码**（来自反汇编）：

```asm
; MiniDumpReadDumpStream - 只有一个间接跳转
0x180001000:  jmp qword_180003158    ; 跳到真函数

; MiniDumpWriteDump - 只有一个间接跳转
0x180001010:  jmp qword_180003160    ; 跳到真函数
```

这两个导出函数是纯粹的"代理"（proxy）——它们不做任何修改，只是把调用转发给真正的系统 DLL。**LeagueClient.exe 完全不知道自己在用伪造的 DLL。**

### 14.4 关键全局变量

| 变量 | 地址 | 初始值 | 用途 |
|------|------|--------|------|
| qword_180003158 | 0x180003158 | 0x0 | 真 MiniDumpReadDumpStream 指针 |
| qword_180003160 | 0x180003160 | 0x0 | 真 MiniDumpWriteDump 指针 |
| qword_180003170 | 0x180003170 | 0x0 | flyinput.dll 模块句柄 |

### 14.5 为什么选择 dbgcore.dll

| 优势 | 说明 |
|------|------|
| **系统 DLL** | 是 Windows 正式组件，不在 KnownDLLs 注册表中（不被强制从 System32 加载） |
| **导出少** | 仅 2 个实质导出函数，转发实现极简 |
| **无防护** | 不被反作弊系统监控（不像 d3d9.dll、dxgi.dll 等常见劫持目标） |
| **加载时机** | LeagueClient.exe 启动时自动加载，无需额外触发 |
| **隐蔽性** | 崩溃转储功能极少被使用，转发调用几乎不会发生 |

### 14.6 与其他劫持方式的对比

| 劫持方式 | 常见目标 DLL | 被反作弊监控 | 实现复杂度 |
|---------|-------------|------------|-----------|
| **dbgcore.dll 劫持** | dbgcore.dll | 极低 | 简单（2 个导出） |
| DirectX 劫持 | d3d9.dll/dxgi.dll | 高 | 中等 |
| 输入法劫持 | version.dll/msimg32.dll | 中 | 简单 |
| COM 劫持 | 注册表 CLSID | 中 | 复杂 |
| 计划任务劫持 | 任务计划程序 | 低 | 中等 |

**结论**：dbgcore.dll 劫持是作者精心选择的攻击面，在隐蔽性和实现复杂度之间取得了最佳平衡。

## 15. 游戏更新后如何更新 DLL 继续使用

### 15.1 核心问题：所有偏移都是硬编码的

flyinput.dll 中**没有任何特征码扫描（pattern scan）或自动更新机制**。所有 51 个游戏内存偏移都是硬编码的 RIP-relative 常量。

**证据**（来自 round4_memory_offsets.txt）：

```
// 对象列表指针 - 硬编码在 DLL 中
qword_180019380 = *((_QWORD *)_RAX + 4010496);   // 防御塔列表
qword_180019378 = *((_QWORD *)_RAX + 4013091);   // 英雄列表
qword_180019388 = *((_QWORD *)_RAX + 4014672);   // 小兵列表
qword_180019390 = *((_QWORD *)_RAX + 4016994);   // 野怪列表

// 游戏全局偏移 - 硬编码
0x01E99000  // 视距/缩放数据
0x01EA1280  // 对象列表
0x01EA12B0  // 对象列表指针
0x01EAD040  // 游戏主状态指针
0x01EDB0A8  // 全局数据
0x01F65480  // 全局数据

// 对象结构体偏移 - 硬编码
+0x0C     // 对象类型
+0x1080   // 当前 HP
+0x1710   // 基础攻击力
+0x360    // 攻击范围
+0x6F8    // 碰撞半径
```

### 15.2 游戏更新影响哪些偏移

每次 LOL 版本更新，以下偏移都可能变化：

| 类别 | 偏移示例 | 变化频率 | 影响 |
|------|---------|---------|------|
| **全局指针** | 0x1EAD040, 0x1EA1280 | 每次更新 | 程序无法找到任何游戏数据 |
| **对象列表偏移** | +4010496, +4013091, +4014672, +4016994 | 每次更新 | 无法遍历英雄/小兵/野怪/防御塔 |
| **对象结构体偏移** | +0x1080(HP), +0x1710(AD), +0x360(范围) | 偶尔变化 | 读取错误数据或崩溃 |
| **技能列表偏移** | +0x2E90, +0x3C10 | 偶尔变化 | 自动技能失效 |
| **功能标志位** | +0x14A0 | 偶尔变化 | 开关失效 |

### 15.3 更新 DLL 的完整步骤

**前提**：你已经有解壳后的 flyinput.dll（96,256 bytes），以及 IDA Pro 或 x64dbg 可用于分析新版本游戏。

```
步骤 1: 获取新版本游戏的内存偏移
├─ 用 IDA Pro 分析新版本 League of Legends.exe
├─ 或用 ReClass.NET/Cheat Engine 动态找偏移
├─ 或用 Lmeme 等开源项目的特征码表
└─ 记录所有 51 个偏移的新值

步骤 2: 在解壳的 flyinput.dll 中定位所有偏移
├─ 在 IDA 中搜索所有硬编码的立即数
├─ 每个偏移在 DLL 中以 RIP-relative 方式存储
├─ 例如: mov rax, [rcx+0x1EAD040] 在文件中是 48 8B 81 40 D0 EA 01
└─ 列出所有需要修改的文件偏移和原始值

步骤 3: 批量修改偏移
├─ 编写 Python 脚本批量替换
├─ 每个偏移修改 4 字节（32 位立即数）
└─ 验证修改后的 DLL 结构完整性

步骤 4: 重新 UPX 加壳
├─ upx --ultra-brute -f -o flyinput_new.dll flyinput_unpacked.dll
└─ 确保大小 ≤ 31,232 bytes（PE2 槽位限制）

步骤 5: 替换到 MJTVPFeD.dat
├─ 用新的 flyinput.dll 替换 PE2 @ 0xF1D0
└─ 尾部补 0 填充到 31,232 bytes

步骤 6: RC4 重新加密
├─ 用 rc4_keystream.bin XOR 加密修改后的明文
└─ 密钥流不变（密钥=硬件指纹，未变）

步骤 7: 回写到 16.13.exe
├─ 拼接: 原始前 0x121000 bytes + 8 bytes header + 新加密数据
└─ 验证文件大小 = 1,297,416 bytes
```

### 15.6 特征码扫描（推荐实现）

如果要在 flyinput.dll 中增加自动更新能力，可以实现特征码扫描：

```cpp
// 示例: 通过特征码自动定位 ObjectManager
uintptr_t FindPattern(uintptr_t base, size_t size, const char* pattern, const char* mask) {
    for (size_t i = 0; i < size - strlen(mask); i++) {
        bool found = true;
        for (size_t j = 0; j < strlen(mask); j++) {
            if (mask[j] != '?' && *(BYTE*)(base + i + j) != (BYTE)pattern[j]) {
                found = false;
                break;
            }
        }
        if (found) return base + i;
    }
    return 0;
}

// 典型的 ObjectManager 特征码
// 48 8B 05 ? ? ? ? 48 8B 48 08 48 85 C9 74 07
uintptr_t objectManager = FindPattern(base, size,
    "\x48\x8B\x05\x00\x00\x00\x00\x48\x8B\x48\x08\x48\x85\xC9\x74\x07",
    "xxx????xxxxxxxxx");
```

**注意**：当前 flyinput.dll 没有使用特征码扫描，添加此功能需要修改 DLL 代码逻辑（从 "读硬编码偏移" 改为 "特征码扫描 + 缓存结果"）。

### 15.7 实战建议

1. **每次游戏更新前备份**：保存当前工作的 16.13.exe 和 flyinput.dll
2. **关注 Lmeme 更新**：Lmeme 项目通常会在游戏更新后 1-3 天内更新偏移表
3. **优先更新全局偏移**：0x1EAD040、0x1EA1280 等是"根偏移"，找对就能恢复大部分功能
4. **对象结构体偏移变化少**：+0x1080(HP)、+0x1710(AD) 等很少变化，通常不需要更新
5. **测试环境**：在训练模式测试，确认功能正常后再用于实战
