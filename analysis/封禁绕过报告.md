## 三、登录流程逆向分析

### 3.1 整体架构

```
用户点击"登录"按钮
       ↓
按钮处理函数 sub_416A8E (0x416A8E - 0x416C53)
       ↓
调用登录 API sub_4141FA (0x4141FA - 0x414446)
       ↓
   ┌───────────────────────────┐
   │ 返回 0 (失败)              │ 返回 1 (成功)
   ↓                           ↓
跳到失败路径 0x416BCD          调用 sub_416C53 解析用户数据
       ↓                           ↓
显示错误消息框                  显示登录时间
```

### 3.2 sub_4141FA 登录 API 函数详解

**函数签名**: `int __stdcall sub_4141FA(_DWORD *a1_account, _DWORD *a2_password, int a3)`

**功能**: 发送登录请求到服务器,验证响应,保存账号密码

**关键流程**:

```c
// 1. 构造请求字符串
request = sub_40109D("login|account=", *a1, "|account_pass=", *a2);
// 例如: "login|account=Uayj5ffxmYk9|account_pass=xxx"

// 2. 发送 HTTP 请求
response = sub_4098F9(&request);
dword_50F928->vtable[76/4](&dword_50F928, &response, ...);  // HTTP POST

// 3. 检查响应码是否为 "200"
if (sub_40208D(*dword_50F92C, "200")) {
    // 响应不是 200 → 返回 0 (失败)
    return 0;  // ← 封禁账户会走这里
}
// 注意: sub_40208D 返回非零表示"不相等",所以 if 成立时是失败

// 4. 保存账号密码到全局结构
*dword_50F954 = *a1;          // 保存账号到 +0x00
*(dword_50F954 + 4) = *a2;   // 保存密码到 +0x04

// 5. 解析并保存 loginTime, loginCookie
dword_50F954[40] = json_get("result.loginTime");
dword_50F954[76] = json_get("result.loginCookie");

// 6. 返回 1 (成功)
return 1;
```

### 3.3 关键发现:封禁检测是服务器端

**重要结论**: 程序本地**没有**任何"封禁"字符串或本地封禁检查逻辑。

通过搜索整个二进制文件:
- 搜索 "封禁" → 未找到
- 搜索 "banned" → 未找到
- 搜索 "opening" 字段 → 仅在 `sub_416C53` 中被存储,从未被读取检查

**封禁检测机制**:
1. 服务器收到登录请求后,检查账户状态
2. 如果账户被封禁,服务器返回**非 200 状态码**(如 403)
3. 客户端 `sub_4141FA` 检查到非 200,返回 0
4. 按钮处理函数收到返回值 0,跳到失败路径
5. 调用 `sub_412954` 获取错误消息,显示消息框

**用户看到的"账户已被封禁"消息实际来自服务器响应体**,而非本地字符串。

## 四、Patch 方案

### 4.1 Patch 思路

由于封禁检测在服务器端(返回非 200 状态码),本地 patch 的目标是:

**让客户端忽略服务器返回的状态码,强制走"登录成功"路径。**

具体方法:
1. **Patch 1**: 在 `sub_4141FA` 中,跳过 "200" 状态码检查,直接执行保存账号密码的代码
2. **Patch 2**: 在按钮处理函数中,跳过 `sub_4141FA` 返回值检查,直接调用 `sub_416C53`

### 4.2 Patch 1: sub_4141FA "200" 检查跳转

**原始代码** (偏移 0x142CB, VA 0x4142CB):
```asm
0x4142C8: cmp     eax, 0           ; 83 F8 00
0x4142CB: jz      loc_4142DB        ; 0F 84 0A 00 00 00  ← 条件跳转
0x4142D1: mov     eax, 0            ; B8 00 00 00 00     ← 失败:返回 0
0x4142D6: jmp     loc_414440        ; E9 65 01 00 00      ← 跳到函数结尾
0x4142DB: mov     ebx, dword_50F954; 8B 1D 54 F9 50 00   ← 成功:保存账号
```

**逻辑分析**:
- `sub_40208D` 比较字符串,返回非零表示"不相等"
- `cmp eax, 0` 后 `jz` 表示:如果字符串相等(返回 0)则跳到 0x4142DB (成功路径)
- 如果不相等(非 200),继续执行 `mov eax, 0` 返回失败

**Patch 内容**:
将 `jz loc_4142DB` (条件跳转) 改为 `jmp loc_4142DB` (无条件跳转),无论比较结果如何都走成功路径。

**字节修改**:
```
原始: 0F 84 0A 00 00 00    (jz rel32=0x0A, 6字节)
修改: E9 0B 00 00 00 90    (jmp rel32=0x0B, 5字节 + nop, 6字节)
```

**关键计算**:
- `jz` 指令长度 6 字节,目标 = 0x4142CB + 6 + 0x0A = 0x4142DB ✓
- `jmp` 指令长度 5 字节,目标 = 0x4142CB + 5 + rel32
- 要跳到 0x4142DB: rel32 = 0x4142DB - 0x4142CB - 5 = 0x0B = 11

> **⚠️ 关键陷阱**: rel32 必须是 **0x0B** (不是 0x0A)!
> - `jz` 是 6 字节,rel32 基于 6 字节偏移 → 0x0A
> - `jmp` 是 5 字节,rel32 基于 5 字节偏移 → 0x0B
> - 初次 patch 错误使用 0x0A 导致跳到 `jmp` 指令中间字节 0x4142DA,破坏指令对齐,程序崩溃

### 4.3 Patch 2: 按钮处理函数返回值检查

**原始代码** (偏移 0x16B43, VA 0x416B43):
```asm
0x416B3F: cmp     dword ptr [ebp-10h], 0  ; 83 7D F0 00
0x416B43: jz      loc_416BCD              ; 0F 84 84 00 00 00  ← 失败跳转
0x416B49: call    sub_416C53              ; E8 05 01 00 00      ← 成功:解析数据
...
0x416BCD: call    sub_412954              ; E8 82 BF FF FF      ← 失败:显示错误
```

**逻辑分析**:
- `sub_4141FA` 返回值存在 `[ebp-10h]`
- `cmp [ebp-10h], 0` 后 `jz` 表示:如果返回 0 (失败)则跳到错误处理
- 返回 1 (成功)则继续调用 `sub_416C53` 解析用户数据

**Patch 内容**:
将 `jz loc_416BCD` (条件跳转) 改为 NOP (空操作),无论返回值如何都继续执行成功路径。

**字节修改**:
```
原始: 0F 84 84 00 00 00    (jz rel32=0x84, 6字节)
修改: 90 90 90 90 90 90     (nop x 6, 6字节)
```

### 5.1 应用脚本

    (0x142CB,
     bytes.fromhex('0f840a000000'),

     '0x4142CB: jz -> jmp (bypass 200 check)'),

with open(EXE, 'r+b') as f:
    for offset, expected, patched, desc in patches:
        f.seek(offset)
        actual = f.read(len(expected))

        f.seek(offset)
        f.write(patched)

## 八、技术总结

### 8.1 成功原因

1. **VMProtect 保护薄弱**: 仅使用函数级标记,未启用虚拟化,函数体完全可读
2. **封禁检测在服务器端**: 本地无封禁字符串,完全依赖 HTTP 状态码
3. **检查逻辑简单**: 仅 2 个条件跳转需要 patch
4. **无完整性校验**: 程序未对自身进行 hash/CRC 校验

### 8.2 Patch 清单

| # | 文件偏移 | VA | 原始字节 | Patch 字节 | 说明 |
|---|----------|-----|----------|-----------|------|
| 1 | 0x142CB | 0x4142CB | `0F 84 0A 00 00 00` | `E9 0B 00 00 00 90` | jz→jmp 绕过 "200" 检查 |
| 2 | 0x16B43 | 0x416B43 | `0F 84 84 00 00 00` | `90 90 90 90 90 90` | jz→NOP 绕过返回值检查 |

### 8.3 关键教训

1. **rel32 偏移计算**: jz (6字节) 转 jmp (5字节) 时,rel32 必须加 1,否则跳转目标错位
2. **32位程序 UI 自动化**: pywinauto 应使用 `class_name()` 而非 `control_type` 识别控件
3. **服务器端检测无法完全绕过**: 本 patch 仅绕过客户端检查,服务器日志仍会记录封禁账户登录尝试
